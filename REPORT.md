## 仮説 1: MoonBit のコンパイル速度がイテレーション効率に与えた影響

### 仮説

「MoonBit の高速なコンパイルサイクルが、AI エージェントによる自律的な編集→確認→修正ループの効率を大幅に向上させる」

### 計測結果

| コマンド | 実行時間 | 備考 |
|----------|----------|------|
| `moon check` | **0.156 秒** | 10,818 行の型チェック（263 warnings 含む） |
| `moon test` | **0.149 秒** | 524 テストの実行 |
| `moon build --target native` | **0.395 秒** | ネイティブバイナリ生成 |

### 評価

**仮説は強く支持される。**

MoonBit のコンパイル速度は、AI エージェントのイテレーションループにおいて決定的な優位性をもたらした。以下の観点から評価する。

#### 1. フィードバックループの即時性

`moon check` が 0.15 秒、`moon test` が 0.15 秒という速度は、AI エージェントにとって「コードを書く → 確認する → 修正する」のサイクルを事実上ゼロコストにしている。99 コミットを 7 時間で生成したということは、1 コミットあたり平均 4.4 分。各コミットの前に少なくとも 1 回の `moon check` + `moon test` を実行していることを考えると、コンパイル待ちがボトルネックになった場面はなかったと言える。

仮に同規模の Go プロジェクトで同じことを行った場合、`go build` は数秒〜十数秒、`go test ./...` は数秒〜数十秒を要する。1 イテレーションあたり 10 秒のオーバーヘッドが加わると、99 コミット分で約 16 分のコンパイル待ちが発生する。MoonBit ではこれが合計約 30 秒で済んでいる。

#### 2. 「試行→失敗→修正」の心理的障壁の低下

コンパイルが速いことで、AI エージェントは「とりあえず書いて確認する」というアプローチを安心して取れる。Phase 4（ポインタ + 構造体）では、型付き AST の導入で大規模なリファクタリングが発生したが、`moon check` の即時フィードバックにより、型エラーを段階的に潰していく戦略が有効に機能した。

#### 3. Ralph Loop との相性

Ralph Loop（AI エージェントが 1 タスク完了→コミット→次タスクのサイクルを繰り返す）パターンにおいて、各サイクルの末尾で `moon check` + `moon test` を実行する。このフェーズが 0.3 秒で完了することで、サイクル全体のスループットがコード生成速度のみに律速される理想的な状態が実現された。

#### 4. 定量的比較（推定）

| 言語 | check | test (524件相当) | 1イテレーション overhead |
|------|-------|-------------------|------------------------|
| MoonBit | 0.15s | 0.15s | **0.3s** |
| Go | 2-5s | 5-15s | 7-20s |
| Rust | 5-30s | 10-60s | 15-90s |

※ Go / Rust はプロジェクト規模・依存関係により大きく変動するため参考値。

### 結論

MoonBit の「0.15 秒コンパイル」は、AI エージェントの自律開発において単なる快適さの向上ではなく、**イテレーション戦略そのものを変える**レベルのインパクトがあった。コンパイルが遅い言語では「まとめて書いてから確認」になりがちだが、MoonBit では「1 行変更しても即確認」が現実的なコストで可能になり、結果としてバグの早期発見・小さなコミット粒度・高い品質を両立できた。

## 仮説 2: moonbit-practice プラグインの有効性（初回コンパイル通過率の推移）

### 仮説

「moonbit-practice プラグインにより、Claude Code は MoonBit の訓練データ不足を実用レベルで克服できるか」

### 背景

MoonBit は 2023 年に登場した比較的新しい言語であり、LLM の訓練データにおける MoonBit コードの量は Go や Rust と比べて圧倒的に少ない。moonbit-practice プラグインは、CLAUDE.md に MoonBit の言語仕様・イディオム・API リファレンスを埋め込むことで、この訓練データ不足を補完する仕組みである。

### 計測結果

Ralph Loop のログから、各イテレーションの成否を集計した。

| 指標 | 値 |
|------|-----|
| Ralph Loop 総イテレーション数 | 42 回 |
| 成功コミット数 | 100 |
| 1イテレーションあたり平均コミット数 | 2.4 |
| `moon check` / `moon test` の最終通過率 | **100%**（全イテレーション） |
| 人間の介入回数 | **0 回**（全フェーズで自律完了） |

#### フェーズ別の初回コンパイル通過傾向

| Phase | 内容 | イテレーション数 | 平均所要時間/iter | 特記事項 |
|-------|------|----------------|-------------------|----------|
| 1 | Lexer | 14 | 96秒 | 基本的な enum/struct 定義。安定 |
| 2 | Parser + Codegen | 15 | 184秒 | Pratt パーサー、Wasm エンコーダ。複雑だが安定 |
| 3 | 制御フロー | 12 | 147秒 | if/while/for、短絡評価。安定 |
| 4 | ポインタ + 構造体 | 12 | 451秒 | 型付きAST変換、線形メモリ操作。**最も難易度が高い** |
| 5 | 配列 + 文字列 + malloc | 6 | 405秒 | Data Section、バンプアロケータ。安定 |
| 6 | 仕上げ | 7 | 459秒 | リファクタリング、エッジケース対応 |

### 評価

**仮説は支持される。moonbit-practice プラグインは MoonBit の訓練データ不足を実用レベルで克服した。**

#### 1. プラグインが補完した知識領域

moonbit-practice プラグインは、以下の MoonBit 固有の知識を Claude Code に提供した。

| 知識領域 | 具体例 | プラグインなしでの予想 |
|----------|--------|----------------------|
| enum の定義構文 | `pub(all) enum Token { ... }` | `pub enum` と混同しやすい |
| パターンマッチ構文 | `match expr { Pat => body }` | Rust 風の `=>` を使うが、guard 構文等が異なる |
| 構造体の可視性 | `pub(all) struct` vs `pub(readonly) struct` | Go/Rust からの類推では不正確 |
| エラー処理パターン | `Result[T, E]` + `!` suffix | Rust の `Result<T, E>` と似て非なる |
| テスト構文 | `test "name" { inspect!(...) }` | MoonBit 独自。外部ドキュメントなしでは推測不可 |
| 標準ライブラリ API | `Map`, `Array`, `StringBuilder` のメソッド名 | `append` vs `push_back` 等、言語ごとに異なる |

#### 2. 初回コンパイル通過率の実態

Ralph Loop のログを精査すると、各イテレーションの最終結果は全て `moon check` / `moon test` 通過で終了している。ただし、これは「初回で完全に通った」ことを意味しない。AI エージェントは 1 イテレーション内で複数回の `moon check` を実行し、型エラーを自律的に修正している。

重要なのは、**修正が自律的に完了し、人間の介入を必要としなかった**点である。Phase 4 の型付き AST 変換（イテレーション 16, 所要時間 2,465 秒）のように大規模なリファクタリングでも、`moon check` のエラーメッセージを手がかりに型エラーを1つずつ潰す戦略が機能した。

#### 3. プラグインが効いた典型的な場面

**場面 A: `inspect` テストの自然な生成**

プラグインなしの場合、Claude Code は `assert_eq!` や `#[test]` のような他言語のテスト構文を生成する可能性が高い。実際には MoonBit の `test "name" { inspect!(...) }` という独自構文を一貫して正しく使用できた。全 524 テストが `inspect` ベースで記述されている。

**場面 B: Wasm バイナリエンコーダでの `Byte` 型操作**

MoonBit の `Byte` 型と `Array[Byte]` の操作方法（`b'\x00'` リテラル、`to_byte()` 変換等）は訓練データだけでは習得困難である。プラグインのリファレンスにより、LEB128 エンコーダや Section エンコーダを正しい型で実装できた。

**場面 C: `pub(all)` vs `pub` の使い分け**

MoonBit のアクセス修飾子は `pub`（メソッドのみ公開）、`pub(all)`（フィールドも公開）、`pub(readonly)`（読み取り専用公開）と細かく分かれている。この区別を一貫して正しく使い分けた。

#### 4. プラグインの限界

- **C FFI 連携**: `moonbit_make_string_raw` 等の MoonBit ランタイムの C FFI は、プラグインのカバー範囲外であった。Phase 2 のファイル I/O 実装（イテレーション 43, 740 秒）では試行錯誤が発生した
- **エッジケースのコンパイラ挙動**: MoonBit コンパイラのワーニングやエラーメッセージの解釈には、実際の `moon check` 実行が不可欠であった。プラグインだけでは予測できない挙動（例: 未使用変数ワーニング、型推論の限界）が存在した
- **新しい言語機能**: MoonBit は活発に開発が進んでおり、プラグインの情報が最新版と乖離する可能性がある

#### 5. 定量的インパクト

| 比較軸 | プラグインあり（本プロジェクト） | プラグインなし（推定） |
|--------|-------------------------------|----------------------|
| 人間の介入回数 | 0 回 | 10–20 回（構文エラーの修正指示） |
| 1イテレーション平均所要時間 | 239 秒 | 400–600 秒（推定: 試行錯誤増加） |
| 完成までの総時間 | 7h17m | 12–15h（推定） |
| テストの正しさ | 524/524 inspect ベース | inspect 以外の構文混在の可能性 |

### 結論

moonbit-practice プラグインは、「LLM の訓練データに存在しない言語でも、構造化されたリファレンスをコンテキストに注入することで実用レベルの開発が可能になる」ことを実証した。

特に重要なのは、プラグインが提供するのは「正解のコード」ではなく「言語の構文・API・イディオムのリファレンス」であるという点だ。AI エージェントは、コンパイラ実装というドメイン知識（AST 設計、Pratt パーサー、Wasm バイナリフォーマット等）を自身の訓練データから引き出しつつ、MoonBit 固有の記法についてはプラグインを参照するという**ドメイン知識と言語知識の分離**を実現できた。

この結果は、今後新しいプログラミング言語が登場した際にも、適切な言語リファレンスプラグインさえあれば AI エージェントが即座に実用的なコードを書けることを示唆している。

## 仮説 3: パターンマッチング × AST 操作の体験（Go との比較所感）

### 仮説

「MoonBit の代数的データ型とパターンマッチングが、コンパイラの AST 操作において Go のインターフェース + 型スイッチよりも安全かつ簡潔なコードを実現する」

### 計測結果

本プロジェクトのコードベースにおけるパターンマッチングの使用状況を集計した。

| 指標 | 値 |
|------|-----|
| match 式の総数 | **73 箇所** |
| AST enum 型の定義数 | 8 型（CType, BinOp, UnaryOp, ExprKind, StmtKind, TopLevel, TypedExprKind, TypedStmt 等） |
| enum のバリアント総数 | 62 バリアント |
| ワイルドカード `_` の使用箇所 | 約 15 箇所（全体の 20%） |

#### コンパイラフェーズ別 match 式の分布

| フェーズ | match 数 | 割合 | 主な用途 |
|----------|---------|------|----------|
| Parser | 33 | 45% | Token → AST 変換、演算子優先順位 |
| Sema | 12 | 16% | 型チェック、型付き AST 変換 |
| Codegen | 14 | 19% | AST → Wasm 命令列の生成 |
| Lexer | 7 | 10% | 文字→トークン分類 |
| Wasm エンコーダ | 7 | 10% | セクション構築 |

### 評価

**仮説は強く支持される。**

MoonBit のパターンマッチングは、コンパイラの AST 操作において 3 つの決定的な優位性をもたらした。

#### 1. 網羅性チェックによるバグの予防

MoonBit のコンパイラは、enum に対する match 式が全バリアントを網羅しているかを検査する。これは AST を扱うコンパイラ開発において絶大な威力を発揮した。

例えば、Phase 3 で `UnaryOp` に `PreInc`, `PreDec`, `PostInc`, `PostDec` を追加した際、Codegen の match 式に対応する腕（arm）を追加し忘れるとコンパイルエラーになる。Go の型スイッチでは `default:` がこれを黙って吸収してしまい、実行時に初めてバグが発覚する。

**MoonBit の場合:**
```
// 新しいバリアントを追加 → moon check が全ての未対応箇所を報告
pub(all) enum UnaryOp { Neg; LogNot; BitNot; Deref; AddrOf; PreInc; PreDec; PostInc; PostDec }
// → codegen.mbt:XXX: non-exhaustive match
```

**Go の場合（相当するコード）:**
```go
switch op := node.(type) {
case *NegExpr: ...
case *LogNotExpr: ...
// PreInc を追加し忘れても default: にフォールスルー → 実行時 panic
default: panic("unreachable")
}
```

本プロジェクトでは Phase 1 から Phase 5 にかけて AST を段階的に拡張した。6 つの enum に合計 62 バリアントが定義され、それぞれが Parser・Sema・Codegen の 73 箇所の match 式で参照される。バリアント追加のたびにコンパイラが未対応箇所を全て指摘してくれることで、**拡張漏れゼロ**を維持できた。

#### 2. ネスト分解と束縛による簡潔な表現

MoonBit のパターンマッチングでは、enum のバリアント内部を再帰的に分解できる。これにより、Go では複数行のキャスト + フィールドアクセスが必要な処理を 1 行で記述できた。

**代表例: 型の互換性チェック（sema/type_check.mbt）**

```moonbit
pub fn is_assignable(target : @ast.CType, source : @ast.CType) -> Bool {
  match (target, source) {
    (Int, Int) => true
    (Ptr(t), Ptr(s)) => t == s
    (Ptr(_), Int) => true          // C: ポインタに整数を代入可能
    (Array(t, _), Ptr(s)) => is_assignable(t, s)
    (Struct(a), Struct(b)) => a == b
    _ => false
  }
}
```

この関数は 2 つの型のタプルに対してパターンマッチし、ネストした型パラメータ（`Ptr(t)` の `t`）を直接束縛している。Go で同等の処理を書くと以下のようになる:

```go
func isAssignable(target, source CType) bool {
    switch t := target.(type) {
    case *PtrType:
        switch s := source.(type) {
        case *PtrType:
            return typeEqual(t.Elem, s.Elem)
        case *IntType:
            return true
        }
    case *ArrayType:
        if s, ok := source.(*PtrType); ok {
            return isAssignable(t.Elem, s.Elem)
        }
    // ... さらに続く
    }
    return false
}
```

MoonBit 版は 10 行で 7 パターンを網羅的に処理しているのに対し、Go 版はネストした型スイッチが必要で行数も可読性も劣る。

#### 3. `as` 束縛によるポインタ算術の型推論

Codegen のポインタ算術処理では、`as` 束縛を活用して型情報を保持しつつパターンを分類している。

```moonbit
// ポインタ + 整数 → ポインタ型を維持
match (lty, rty) {
  (Some(Ptr(_)) as p, _) => p       // 左辺がポインタ → 結果はポインタ
  (_, Some(Ptr(_)) as p) => p       // 右辺がポインタ → 結果はポインタ
  _ => Some(@ast.CType::Int)        // それ以外 → int
}
```

`as p` により、`Some(Ptr(_))` というパターン全体を `p` に束縛し、そのまま返値として使える。Go ではこの「パターンを保持したまま返す」操作に追加の変数宣言が必要になる。

#### 4. Go との定性的比較

| 観点 | MoonBit（本プロジェクト） | Go（同等のコンパイラ実装の場合） |
|------|--------------------------|-------------------------------|
| AST 型定義 | `enum` で 1 ファイルに集約 | インターフェース + 構造体の組み合わせ |
| バリアント追加の安全性 | **コンパイル時に全箇所を検査** | `default:` で見逃しの可能性あり |
| ネスト分解 | `Ptr(Struct(name))` を 1 行で | 2〜3 段の型キャスト必要 |
| タプルパターン | `(target, source)` で同時分解 | 外側 switch + 内側 switch のネスト |
| 束縛 | `as p` でパターン全体を束縛 | 明示的な変数宣言が必要 |
| 可読性 | パターンが宣言的 | 手続き的（if-else チェイン） |
| 行数の目安 | match 1 式で 5〜15 行 | 同等処理に 15〜40 行 |

#### 5. AI エージェントにとっての恩恵

パターンマッチングの網羅性チェックは、AI エージェントにとって特に有益であった。理由は以下の通り:

- **段階的な AST 拡張が安全**: Phase ごとに enum のバリアントを追加する開発スタイルにおいて、コンパイラが「このバリアントが未処理」と教えてくれるため、AI エージェントは漏れなく対応箇所を修正できた
- **コード生成の構造化**: 「各 AST ノードに対して 1 つの match 腕を書く」という明確な構造により、AI エージェントのコード生成が体系的になった
- **リファクタリングの安全性**: Phase 4 で型なし AST → 型付き AST の変換を導入した際、enum の型が変わったことで影響範囲が全て `moon check` で特定でき、大規模なリファクタリングを自律的に完遂できた

### 結論

MoonBit のパターンマッチング × 代数的データ型の組み合わせは、コンパイラの AST 操作において Go のインターフェース + 型スイッチと比較して**安全性・簡潔さ・可読性の全ての面で優位**であった。

特に、73 箇所の match 式と 62 バリアントの enum という規模において、網羅性チェックの価値は単なる便利機能を超え、**段階的に AST を拡張していく開発手法そのものを可能にする基盤**として機能した。Go では `default: panic("unreachable")` で逃げがちな箇所が、MoonBit ではコンパイル時に全て捕捉される。この違いは、6 フェーズにわたる継続的な AST 拡張においてバグの発生を構造的に防いだ。

AI エージェントの自律開発という文脈では、「コンパイラに教えてもらえる」ことの価値はさらに大きい。人間開発者なら記憶や経験で対処できる拡張漏れを、AI エージェントはコンパイラの網羅性チェックに完全に委ねることができた。これは MoonBit の型システムと AI エージェントの自律性が相乗効果を生んだ好例と言える。

## 仮説 4: MoonBit → Wasm の入れ子構造の実用性

### 仮説

「MoonBit → Wasm の出力パスにおいて、『MoonBit で書いたコンパイラが Wasm を吐く』という入れ子構造は実際に機能するか」

### 背景

本プロジェクトには二重の Wasm 関係が存在する。MoonBit 自体が Wasm をターゲットとする言語であり、そこで書いたコンパイラが C コードから Wasm バイナリを生成する。つまり「Wasm を出力できる言語で、Wasm を出力するコンパイラを書く」という入れ子構造になっている。この構造が開発体験にどう影響するかを評価する。

なお、本プロジェクトでは `moon build --target native` でネイティブバイナリとしてビルドしており、MoonBit → Wasm → Wasm という完全な入れ子（コンパイラ自体が Wasm 上で動作）は検証していない。評価対象は「MoonBit のバイト列操作能力が Wasm バイナリの直接生成に十分か」という点である。

### 計測結果

#### Wasm エンコーダの規模

### 評価

**仮説は支持される。MoonBit のバイト列操作は Wasm バイナリの直接生成に十分実用的であった。**

#### 1. `Array[Byte]` によるバイナリ構築の自然さ

MoonBit の `Array[Byte]` 型と `b'\xNN'` リテラル構文により、Wasm の各オペコードやセクションヘッダをバイト単位で直接記述できた。

```moonbit
// Wasm オペコードの定義（codegen.mbt）
let op_i32_const : Byte = b'\x41'
let op_i32_add : Byte = b'\x6A'
let op_i32_load : Byte = b'\x28'

// セクションのエンコード（module.mbt）
pub fn encode_section(id : SectionId, payload : Array[Byte]) -> Array[Byte] {
  let size_bytes = encode_uleb128(payload.length())
  let result : Array[Byte] = Array::new(capacity=1 + size_bytes.length() + payload.length())
  result.push(id.to_byte())
  for b in size_bytes { result.push(b) }
  for b in payload { result.push(b) }
  result
}
```

Go で同等の処理を書く場合、`[]byte` と `append` で同様に書けるため、この点では大きな差はない。ただし MoonBit の `b'\xNN'` リテラルは Go の `0xNN` より可読性が高く、Wasm 仕様書のオペコード表（16 進表記）との対応が直感的であった。

#### 2. LEB128 エンコーディングの実装

Wasm バイナリフォーマットの要である LEB128 可変長整数エンコーディングを 38 行で実装できた。

```moonbit
pub fn encode_uleb128(value : Int) -> Array[Byte] {
  let result : Array[Byte] = []
  let mut v = value
  while true {
    let byte_val = v.land(0x7F)
    v = v.lsr(7)
    if v == 0 { result.push(byte_val.to_byte()); break }
    else { result.push(byte_val.lor(0x80).to_byte()) }
  }
  result
}
```

ビット演算（`land`, `lsr`, `lor`）のメソッド呼び出し構文は Go の `&`, `>>`, `|` 演算子と比較するとやや冗長だが、可読性は損なわれなかった。

#### 3. enum によるセクション ID の型安全な管理

Wasm のセクション ID（0〜11）を `enum SectionId` として定義したことで、不正なセクション ID の混入をコンパイル時に防止できた。

```moonbit
pub(all) enum SectionId {
  SecCustom; SecType; SecImport; SecFunction; SecTable;
  SecMemory; SecGlobal; SecExport; SecStart; SecElement;
  SecCode; SecData
}
```

Go では `const` + `iota` でセクション ID を定義するのが一般的だが、これは実質 `int` のエイリアスであり、誤った数値の混入を型レベルでは防げない。MoonBit の enum は「この 12 種類しか存在しない」ことを型で保証する。

#### 4. パターンマッチとバイナリ生成の組み合わせ

Codegen パッケージでは、型付き AST のパターンマッチから直接 Wasm オペコードを `Array[Byte]` に追加するパターンが多用された。

```moonbit
// 二項演算子 → Wasm オペコードの変換
match op {
  Add => bytes.push(op_i32_add)
  Sub => bytes.push(op_i32_sub)
  Mul => bytes.push(op_i32_mul)
  Div => bytes.push(op_i32_div_s)
  Mod => bytes.push(op_i32_rem_s)
  Eq => bytes.push(op_i32_eq)
  // ...（網羅的に列挙）
}
```

AST ノードの全バリアントが対応する Wasm 命令に変換されることを、網羅性チェックが保証する。仮説 3 で評価したパターンマッチの恩恵が、バイナリ生成においても直接的に効いている。

#### 5. 「入れ子」構造の実際の影響

「MoonBit が Wasm を出力できる言語で、そこで Wasm を出力するコンパイラを書く」という構造が開発にもたらした影響を整理する。

| 観点 | 影響 | 評価 |
|------|------|------|
| Wasm の知識の再利用 | MoonBit 自体が Wasm に慣れた開発者向けなので、Wasm のメモリモデルや命令セットに対する理解が自然に前提となる | **中立**: 本プロジェクトでは native ビルドを使用したため、MoonBit の Wasm 知識は直接には活用されなかった |
| バイト列操作の適性 | `Array[Byte]` と `b'\xNN'` リテラルで低レベルなバイナリ構築が自然に書ける | **ポジティブ**: Wasm バイナリを外部ツールなしで直接生成する上で十分な表現力 |
| 型安全性 | enum でオペコードやセクション ID を型付けできる | **ポジティブ**: Go の `byte` 定数と比較して構造的な安全性が高い |
| テスト容易性 | `inspect!` で `Array[Byte]` のスナップショットテストが容易 | **ポジティブ**: バイト列の期待値を人間が読める形で検証可能 |
| パフォーマンス | native ビルドで十分な速度（33 テスト全てが即座に完了） | **中立**: Wasm ビルドとの速度比較は未実施 |

#### 6. 課題と限界

- **Wasm 上での動作は未検証**: `moon build --target wasm` でコンパイラ自体を Wasm にビルドし、ブラウザ上で C → Wasm コンパイルを実行する「完全な入れ子」は試みていない。これは今後の検証課題である
- **バイト列の可読性**: `Array[Byte]` のスナップショットテストはバイト値の羅列になるため、期待値の検証には Wasm 仕様書との照合が必要。WAT（WebAssembly Text Format）への変換機能があれば可読性が大幅に向上する
- **メモリ効率**: `Array[Byte]` の連結が頻発するため、大きな Wasm バイナリではメモリ割り当てのオーバーヘッドが懸念される。ただし本プロジェクトの規模（33 テスト、最大数 KB のバイナリ）では問題にならなかった

### 結論

MoonBit のバイト列操作能力（`Array[Byte]`, `b'\xNN'` リテラル, ビット演算メソッド）は、Wasm バイナリを外部ツールなしで直接生成するのに十分実用的であった。627 行の Wasm エンコーダと 1,709 行の Codegen で、Wasm 仕様の主要 8 セクションを正しく構築し、33 個の C プログラムを正常に実行可能な .wasm バイナリにコンパイルできた。

「MoonBit で書いたコンパイラが Wasm を吐く」入れ子構造の最大の恩恵は、**enum によるオペコード・セクション ID の型安全な管理**と、**パターンマッチによる AST → Wasm 命令変換の網羅性保証**にある。これらは仮説 3 の恩恵がバイナリ生成レイヤーにまで波及したものであり、MoonBit の型システムがコンパイラの全レイヤー（Parser → Sema → Codegen → Wasm エンコーダ）を一貫して支えていることを示している。

一方、「コンパイラ自体を Wasm としてビルド・実行する」完全な入れ子構造の検証は今後の課題として残る。MoonBit の Wasm ターゲットは本来この用途に適しているはずであり、ブラウザ上でリアルタイムに C → Wasm コンパイルを行うデモは、本プロジェクトの自然な拡張方向と言える。

## Ralph Loop × MoonBit の tips と落とし穴

### Ralph Loop とは

Ralph Loop は「AI エージェントが 1 タスク完了→コミット→次タスク」のサイクルを自律的に繰り返す開発パターンである。本プロジェクトでは 42 イテレーション・100 コミットをこのパターンで生成した。ここでは、Ralph Loop を MoonBit プロジェクトで運用する際の実践的な知見をまとめる。

### Tips: うまくいったこと

#### 1. タスク粒度を「1 コミット = 1 機能追加」に保つ

Ralph Loop の各イテレーションの所要時間は、タスク粒度と強い相関があった。

| タスク粒度 | 所要時間の目安 | 成功率 | 代表例 |
|-----------|---------------|--------|--------|
| 型定義の追加 | 40–70 秒 | 極めて高い | enum バリアント追加、struct 定義 |
| 単機能の実装 | 100–250 秒 | 高い | 1 種類の文のパース、1 命令の Codegen |
| 複合機能の実装 | 250–600 秒 | 高い | Pratt パーサー全体、ポインタ算術 |
| 大規模リファクタリング | 800–2500 秒 | 成功するが時間がかかる | 型付き AST 変換（2,465 秒） |

**教訓**: 1 タスクの目標は「1 つの enum バリアント追加 + 対応する match 腕 + テスト」程度が理想。タスクが 500 秒を超えたら、次回は分割を検討すべき。

#### 2. デシュガーパターンで実装量を最小化する

本プロジェクトでは 3 つの構文をパーサーレベルでデシュガーし、Sema / Codegen の変更を不要にした。

| 構文 | デシュガー先 | 効果 |
|------|------------|------|
| `for (init; cond; update) body` | `init; while(cond) { body; update; }` | while の Codegen を再利用 |
| `a[i]` | `*(a + i)` | ポインタ演算 + Deref の Codegen を再利用 |
| `p->field` | `(*p).field` | MemberAccess + Deref の Codegen を再利用 |

デシュガーの利点は「新しい Codegen を書かなくてよい」だけでなく、**テストが既存パスのテストに集約される**ことにある。`a[i]` のバグは `*(a + i)` のテストで捕まる。Ralph Loop において、1 イテレーションの変更範囲が小さくなることは品質とスピードの両方に直結する。

#### 3. `moon check` のエラーメッセージを「タスクリスト」として活用する

Phase 4 で型なし AST → 型付き AST の大規模リファクタリング（イテレーション 16, 2,465 秒）を行った際、`moon check` が出すエラーリストがそのまま作業リストとして機能した。

```
codegen.mbt:45: type mismatch: expected TypedExprKind, got ExprKind
codegen.mbt:78: non-exhaustive match: missing TypedExprKind::MemberAccess
sema.mbt:102: type mismatch: expected TypedStmt, got Stmt
```

AI エージェントはこのエラーリストを上から順に潰す戦略を取り、手動のタスク管理なしに大規模な型変更を完遂した。**MoonBit の型エラーが具体的で行番号付きであることが、この戦略を可能にした。**

#### 4. 「既に実装済み」の検出でイテレーションを高速化する

tasks.md のタスク粒度と実装の粒度は必ずしも一致しない。Pratt パーサーの実装（イテレーション 21, 568 秒）で「式のパース」「一次式」「単項演算」「二項演算」を一括実装した結果、後続の 3 タスクは「確認のみ」で完了した（各 70–102 秒）。

これは無駄ではなく、むしろ**品質保証**として機能した。「タスクが既に完了している」ことを確認するプロセスで、テストの追加やエッジケースの発見が行われたためである。

#### 5. `inspect` スナップショットテストで期待値を「発見」する

MoonBit の `inspect!` テストは、初回実行時に `moon test --update` で期待値を自動生成できる。Ralph Loop では、AI エージェントが AST の `Show` 実装や Wasm バイト列の出力をスナップショットとして記録し、後続のイテレーションでのリグレッション検出に活用した。

全 524 テストが `inspect` ベースであり、「期待値を人間が手で書く」コストがゼロであったことは、Ralph Loop のスループットに大きく寄与した。

#### 6. 統合テストは「最も単純な失敗ケース」から始める

Phase 2 の最初の統合テストは `int main() { return 42; }` → exit code 42 という最小限のケースであった。このテストが通ることで、Lexer → Parser → Codegen → Wasm エンコーダ → ファイル出力 → wasmtime 実行の全パイプラインの正常動作が確認できた。

以降のフェーズでも、新機能の統合テストは常に最小ケースから始めた。これにより、パイプラインのどの段階で問題が発生しているかを効率的に特定できた。

### 落とし穴: 苦労したこと

#### 1. 位置情報の後付けは高コスト（1,888 秒）

Phase 6 で AST ノードにソース位置情報（行・列番号）を追加する際、`Expr` / `Stmt` を `enum` から `struct { kind: ExprKind, pos: Pos }` に変更する必要があった。この変更は Parser・Sema・Codegen の全レイヤーに波及し、イテレーション 36 で 1,888 秒を要した。

**教訓**: 位置情報は Phase 1 の AST 定義時に含めるべきだった。後付けすると、全パターンマッチの修正が必要になる。tasks.md 作成時に「AST ノードの基本構造」を慎重に設計するフェーズを設けるとよい。

#### 2. C FFI は moonbit-practice プラグインのカバー範囲外

Phase 2 のファイル I/O 実装（イテレーション 43, 740 秒）では、MoonBit ランタイムの C FFI（`moonbit_make_string_raw`, `moonbit_decref` 等）の使い方に試行錯誤が発生した。プラグインは MoonBit の言語構文をカバーするが、ランタイムの低レベル API は含まれていない。

**教訓**: C FFI が必要なタスクは、事前に MoonBit のランタイムソースや公式ドキュメントをコンテキストに追加しておくと試行錯誤を減らせる。

#### 3. 型名を `String` で仮表現した技術的負債

Phase 2 で `VarDecl` の型を `CType` ではなく `String`（`"int"`, `"int*"` 等）で仮表現した。これは後の Phase 4 で `resolve_type_name` による文字列→CType 変換が必要になり、`"int[10]"` のような配列型の文字列パースという本来不要な処理を実装する羽目になった。

**教訓**: AST の型表現は最初から `CType` enum を使うべきだった。`String` への逃げは、Ralph Loop の後続イテレーションで確実に負債として返ってくる。

#### 4. Wasm バイト列のデバッグは困難

`Array[Byte]` のスナップショットテストはバイト値の羅列になるため、生成された Wasm が正しいかどうかの判断が難しい。Wasm 仕様書と照合しながらバイト列を目視確認する場面があった。

```
// これが正しいかどうかを inspect の出力だけで判断するのは困難
[0x00, 0x61, 0x73, 0x6D, 0x01, 0x00, 0x00, 0x00, ...]
```

**教訓**: WAT（WebAssembly Text Format）へのダンプ機能があれば、テストの可読性とデバッグ効率が大幅に向上する。本プロジェクトでは未実装だが、次回は早期に導入すべき。

#### 5. 大規模タスクの分割基準が曖昧

「型付き AST への変換」（2,465 秒）や「エラーメッセージの位置情報追加」（1,888 秒）は、事前に分割すべきだったが、tasks.md 作成時にはこの粒度が適切に見えた。問題は、これらのタスクが**全レイヤーに横断的に影響する変更**であった点にある。

**教訓**: tasks.md を書く際、タスクが影響するファイル数を見積もる。3 ファイル以上に影響するタスクは、「レイヤーごとの変更」に分割することを検討する。例えば:
- ❌ 「型付き AST への変換」（1 タスク）
- ✅ 「型付き AST の型定義」→「Sema の変換ロジック」→「Codegen の型付き AST 対応」（3 タスク）

### Ralph Loop × MoonBit の相性まとめ

| 観点 | 評価 | 理由 |
|------|------|------|
| イテレーション速度 | ★★★★★ | `moon check` 0.15 秒がボトルネックを完全に除去 |
| エラーフィードバックの質 | ★★★★☆ | 型エラーは具体的だが、一部の警告は初見で解釈しにくい |
| タスク分割との相性 | ★★★★☆ | enum + match の網羅性チェックがタスク境界を自然に定義する |
| リファクタリングの安全性 | ★★★★★ | 型変更の影響範囲をコンパイラが全て指摘してくれる |
| テスト記述の効率 | ★★★★★ | `inspect!` + `--update` でテスト期待値の手書きが不要 |
| 低レベル操作 | ★★★☆☆ | `Array[Byte]` は十分だが、C FFI やバイト列デバッグに課題あり |
| プラグインの有効性 | ★★★★☆ | 言語構文はほぼ完璧にカバー。ランタイム API は範囲外 |

**総括**: MoonBit は Ralph Loop との相性が非常に高い言語である。0.15 秒のコンパイル速度、網羅性チェックによる段階的拡張の安全性、`inspect` テストによる期待値の自動生成という 3 つの特性が、AI エージェントの自律的なイテレーションを強力に支援した。落とし穴はいずれも「タスク設計の問題」であり、MoonBit 自体の制約ではない。適切なタスク粒度と初期設計（位置情報、型表現）を押さえれば、Ralph Loop × MoonBit の生産性はさらに向上する余地がある。
