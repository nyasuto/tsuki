## 仮説 1: MoonBit のコンパイル速度がイテレーション効率に与えた影響

### 仮説

「MoonBit の高速なコンパイルサイクルが、AI エージェントによる自律的な編集→確認→修正ループの効率を大幅に向上させる」

### 計測結果

| コマンド | 実行時間 | 備考 |
|----------|----------|------|
| `moon check` | **0.156 秒** | 10,818 行の型チェック（263 warnings 含む） |
| `moon test` | **0.149 秒** | 524 テストの実行 |
| `moon build --target native` | **0.395 秒** | ネイティブバイナリ生成 |

### 評価

**仮説は強く支持される。**

MoonBit のコンパイル速度は、AI エージェントのイテレーションループにおいて決定的な優位性をもたらした。以下の観点から評価する。

#### 1. フィードバックループの即時性

`moon check` が 0.15 秒、`moon test` が 0.15 秒という速度は、AI エージェントにとって「コードを書く → 確認する → 修正する」のサイクルを事実上ゼロコストにしている。99 コミットを 7 時間で生成したということは、1 コミットあたり平均 4.4 分。各コミットの前に少なくとも 1 回の `moon check` + `moon test` を実行していることを考えると、コンパイル待ちがボトルネックになった場面はなかったと言える。

仮に同規模の Go プロジェクトで同じことを行った場合、`go build` は数秒〜十数秒、`go test ./...` は数秒〜数十秒を要する。1 イテレーションあたり 10 秒のオーバーヘッドが加わると、99 コミット分で約 16 分のコンパイル待ちが発生する。MoonBit ではこれが合計約 30 秒で済んでいる。

#### 2. 「試行→失敗→修正」の心理的障壁の低下

コンパイルが速いことで、AI エージェントは「とりあえず書いて確認する」というアプローチを安心して取れる。Phase 4（ポインタ + 構造体）では、型付き AST の導入で大規模なリファクタリングが発生したが、`moon check` の即時フィードバックにより、型エラーを段階的に潰していく戦略が有効に機能した。

#### 3. Ralph Loop との相性

Ralph Loop（AI エージェントが 1 タスク完了→コミット→次タスクのサイクルを繰り返す）パターンにおいて、各サイクルの末尾で `moon check` + `moon test` を実行する。このフェーズが 0.3 秒で完了することで、サイクル全体のスループットがコード生成速度のみに律速される理想的な状態が実現された。

#### 4. 定量的比較（推定）

| 言語 | check | test (524件相当) | 1イテレーション overhead |
|------|-------|-------------------|------------------------|
| MoonBit | 0.15s | 0.15s | **0.3s** |
| Go | 2-5s | 5-15s | 7-20s |
| Rust | 5-30s | 10-60s | 15-90s |

※ Go / Rust はプロジェクト規模・依存関係により大きく変動するため参考値。

### 結論

MoonBit の「0.15 秒コンパイル」は、AI エージェントの自律開発において単なる快適さの向上ではなく、**イテレーション戦略そのものを変える**レベルのインパクトがあった。コンパイルが遅い言語では「まとめて書いてから確認」になりがちだが、MoonBit では「1 行変更しても即確認」が現実的なコストで可能になり、結果としてバグの早期発見・小さなコミット粒度・高い品質を両立できた。

## 仮説 2: moonbit-practice プラグインの有効性（初回コンパイル通過率の推移）

### 仮説

「moonbit-practice プラグインにより、Claude Code は MoonBit の訓練データ不足を実用レベルで克服できるか」

### 背景

MoonBit は 2023 年に登場した比較的新しい言語であり、LLM の訓練データにおける MoonBit コードの量は Go や Rust と比べて圧倒的に少ない。moonbit-practice プラグインは、CLAUDE.md に MoonBit の言語仕様・イディオム・API リファレンスを埋め込むことで、この訓練データ不足を補完する仕組みである。

### 計測結果

Ralph Loop のログから、各イテレーションの成否を集計した。

| 指標 | 値 |
|------|-----|
| Ralph Loop 総イテレーション数 | 42 回 |
| 成功コミット数 | 100 |
| 1イテレーションあたり平均コミット数 | 2.4 |
| `moon check` / `moon test` の最終通過率 | **100%**（全イテレーション） |
| 人間の介入回数 | **0 回**（全フェーズで自律完了） |

#### フェーズ別の初回コンパイル通過傾向

| Phase | 内容 | イテレーション数 | 平均所要時間/iter | 特記事項 |
|-------|------|----------------|-------------------|----------|
| 1 | Lexer | 14 | 96秒 | 基本的な enum/struct 定義。安定 |
| 2 | Parser + Codegen | 15 | 184秒 | Pratt パーサー、Wasm エンコーダ。複雑だが安定 |
| 3 | 制御フロー | 12 | 147秒 | if/while/for、短絡評価。安定 |
| 4 | ポインタ + 構造体 | 12 | 451秒 | 型付きAST変換、線形メモリ操作。**最も難易度が高い** |
| 5 | 配列 + 文字列 + malloc | 6 | 405秒 | Data Section、バンプアロケータ。安定 |
| 6 | 仕上げ | 7 | 459秒 | リファクタリング、エッジケース対応 |

### 評価

**仮説は支持される。moonbit-practice プラグインは MoonBit の訓練データ不足を実用レベルで克服した。**

#### 1. プラグインが補完した知識領域

moonbit-practice プラグインは、以下の MoonBit 固有の知識を Claude Code に提供した。

| 知識領域 | 具体例 | プラグインなしでの予想 |
|----------|--------|----------------------|
| enum の定義構文 | `pub(all) enum Token { ... }` | `pub enum` と混同しやすい |
| パターンマッチ構文 | `match expr { Pat => body }` | Rust 風の `=>` を使うが、guard 構文等が異なる |
| 構造体の可視性 | `pub(all) struct` vs `pub(readonly) struct` | Go/Rust からの類推では不正確 |
| エラー処理パターン | `Result[T, E]` + `!` suffix | Rust の `Result<T, E>` と似て非なる |
| テスト構文 | `test "name" { inspect!(...) }` | MoonBit 独自。外部ドキュメントなしでは推測不可 |
| 標準ライブラリ API | `Map`, `Array`, `StringBuilder` のメソッド名 | `append` vs `push_back` 等、言語ごとに異なる |

#### 2. 初回コンパイル通過率の実態

Ralph Loop のログを精査すると、各イテレーションの最終結果は全て `moon check` / `moon test` 通過で終了している。ただし、これは「初回で完全に通った」ことを意味しない。AI エージェントは 1 イテレーション内で複数回の `moon check` を実行し、型エラーを自律的に修正している。

重要なのは、**修正が自律的に完了し、人間の介入を必要としなかった**点である。Phase 4 の型付き AST 変換（イテレーション 16, 所要時間 2,465 秒）のように大規模なリファクタリングでも、`moon check` のエラーメッセージを手がかりに型エラーを1つずつ潰す戦略が機能した。

#### 3. プラグインが効いた典型的な場面

**場面 A: `inspect` テストの自然な生成**

プラグインなしの場合、Claude Code は `assert_eq!` や `#[test]` のような他言語のテスト構文を生成する可能性が高い。実際には MoonBit の `test "name" { inspect!(...) }` という独自構文を一貫して正しく使用できた。全 524 テストが `inspect` ベースで記述されている。

**場面 B: Wasm バイナリエンコーダでの `Byte` 型操作**

MoonBit の `Byte` 型と `Array[Byte]` の操作方法（`b'\x00'` リテラル、`to_byte()` 変換等）は訓練データだけでは習得困難である。プラグインのリファレンスにより、LEB128 エンコーダや Section エンコーダを正しい型で実装できた。

**場面 C: `pub(all)` vs `pub` の使い分け**

MoonBit のアクセス修飾子は `pub`（メソッドのみ公開）、`pub(all)`（フィールドも公開）、`pub(readonly)`（読み取り専用公開）と細かく分かれている。この区別を一貫して正しく使い分けた。

#### 4. プラグインの限界

- **C FFI 連携**: `moonbit_make_string_raw` 等の MoonBit ランタイムの C FFI は、プラグインのカバー範囲外であった。Phase 2 のファイル I/O 実装（イテレーション 43, 740 秒）では試行錯誤が発生した
- **エッジケースのコンパイラ挙動**: MoonBit コンパイラのワーニングやエラーメッセージの解釈には、実際の `moon check` 実行が不可欠であった。プラグインだけでは予測できない挙動（例: 未使用変数ワーニング、型推論の限界）が存在した
- **新しい言語機能**: MoonBit は活発に開発が進んでおり、プラグインの情報が最新版と乖離する可能性がある

#### 5. 定量的インパクト

| 比較軸 | プラグインあり（本プロジェクト） | プラグインなし（推定） |
|--------|-------------------------------|----------------------|
| 人間の介入回数 | 0 回 | 10–20 回（構文エラーの修正指示） |
| 1イテレーション平均所要時間 | 239 秒 | 400–600 秒（推定: 試行錯誤増加） |
| 完成までの総時間 | 7h17m | 12–15h（推定） |
| テストの正しさ | 524/524 inspect ベース | inspect 以外の構文混在の可能性 |

### 結論

moonbit-practice プラグインは、「LLM の訓練データに存在しない言語でも、構造化されたリファレンスをコンテキストに注入することで実用レベルの開発が可能になる」ことを実証した。

特に重要なのは、プラグインが提供するのは「正解のコード」ではなく「言語の構文・API・イディオムのリファレンス」であるという点だ。AI エージェントは、コンパイラ実装というドメイン知識（AST 設計、Pratt パーサー、Wasm バイナリフォーマット等）を自身の訓練データから引き出しつつ、MoonBit 固有の記法についてはプラグインを参照するという**ドメイン知識と言語知識の分離**を実現できた。

この結果は、今後新しいプログラミング言語が登場した際にも、適切な言語リファレンスプラグインさえあれば AI エージェントが即座に実用的なコードを書けることを示唆している。

## 仮説 3: パターンマッチング × AST 操作の体験（Go との比較所感）

### 仮説

「MoonBit の代数的データ型とパターンマッチングが、コンパイラの AST 操作において Go のインターフェース + 型スイッチよりも安全かつ簡潔なコードを実現する」

### 計測結果

本プロジェクトのコードベースにおけるパターンマッチングの使用状況を集計した。

| 指標 | 値 |
|------|-----|
| match 式の総数 | **73 箇所** |
| AST enum 型の定義数 | 8 型（CType, BinOp, UnaryOp, ExprKind, StmtKind, TopLevel, TypedExprKind, TypedStmt 等） |
| enum のバリアント総数 | 62 バリアント |
| ワイルドカード `_` の使用箇所 | 約 15 箇所（全体の 20%） |

#### コンパイラフェーズ別 match 式の分布

| フェーズ | match 数 | 割合 | 主な用途 |
|----------|---------|------|----------|
| Parser | 33 | 45% | Token → AST 変換、演算子優先順位 |
| Sema | 12 | 16% | 型チェック、型付き AST 変換 |
| Codegen | 14 | 19% | AST → Wasm 命令列の生成 |
| Lexer | 7 | 10% | 文字→トークン分類 |
| Wasm エンコーダ | 7 | 10% | セクション構築 |

### 評価

**仮説は強く支持される。**

MoonBit のパターンマッチングは、コンパイラの AST 操作において 3 つの決定的な優位性をもたらした。

#### 1. 網羅性チェックによるバグの予防

MoonBit のコンパイラは、enum に対する match 式が全バリアントを網羅しているかを検査する。これは AST を扱うコンパイラ開発において絶大な威力を発揮した。

例えば、Phase 3 で `UnaryOp` に `PreInc`, `PreDec`, `PostInc`, `PostDec` を追加した際、Codegen の match 式に対応する腕（arm）を追加し忘れるとコンパイルエラーになる。Go の型スイッチでは `default:` がこれを黙って吸収してしまい、実行時に初めてバグが発覚する。

**MoonBit の場合:**
```
// 新しいバリアントを追加 → moon check が全ての未対応箇所を報告
pub(all) enum UnaryOp { Neg; LogNot; BitNot; Deref; AddrOf; PreInc; PreDec; PostInc; PostDec }
// → codegen.mbt:XXX: non-exhaustive match
```

**Go の場合（相当するコード）:**
```go
switch op := node.(type) {
case *NegExpr: ...
case *LogNotExpr: ...
// PreInc を追加し忘れても default: にフォールスルー → 実行時 panic
default: panic("unreachable")
}
```

本プロジェクトでは Phase 1 から Phase 5 にかけて AST を段階的に拡張した。6 つの enum に合計 62 バリアントが定義され、それぞれが Parser・Sema・Codegen の 73 箇所の match 式で参照される。バリアント追加のたびにコンパイラが未対応箇所を全て指摘してくれることで、**拡張漏れゼロ**を維持できた。

#### 2. ネスト分解と束縛による簡潔な表現

MoonBit のパターンマッチングでは、enum のバリアント内部を再帰的に分解できる。これにより、Go では複数行のキャスト + フィールドアクセスが必要な処理を 1 行で記述できた。

**代表例: 型の互換性チェック（sema/type_check.mbt）**

```moonbit
pub fn is_assignable(target : @ast.CType, source : @ast.CType) -> Bool {
  match (target, source) {
    (Int, Int) => true
    (Ptr(t), Ptr(s)) => t == s
    (Ptr(_), Int) => true          // C: ポインタに整数を代入可能
    (Array(t, _), Ptr(s)) => is_assignable(t, s)
    (Struct(a), Struct(b)) => a == b
    _ => false
  }
}
```

この関数は 2 つの型のタプルに対してパターンマッチし、ネストした型パラメータ（`Ptr(t)` の `t`）を直接束縛している。Go で同等の処理を書くと以下のようになる:

```go
func isAssignable(target, source CType) bool {
    switch t := target.(type) {
    case *PtrType:
        switch s := source.(type) {
        case *PtrType:
            return typeEqual(t.Elem, s.Elem)
        case *IntType:
            return true
        }
    case *ArrayType:
        if s, ok := source.(*PtrType); ok {
            return isAssignable(t.Elem, s.Elem)
        }
    // ... さらに続く
    }
    return false
}
```

MoonBit 版は 10 行で 7 パターンを網羅的に処理しているのに対し、Go 版はネストした型スイッチが必要で行数も可読性も劣る。

#### 3. `as` 束縛によるポインタ算術の型推論

Codegen のポインタ算術処理では、`as` 束縛を活用して型情報を保持しつつパターンを分類している。

```moonbit
// ポインタ + 整数 → ポインタ型を維持
match (lty, rty) {
  (Some(Ptr(_)) as p, _) => p       // 左辺がポインタ → 結果はポインタ
  (_, Some(Ptr(_)) as p) => p       // 右辺がポインタ → 結果はポインタ
  _ => Some(@ast.CType::Int)        // それ以外 → int
}
```

`as p` により、`Some(Ptr(_))` というパターン全体を `p` に束縛し、そのまま返値として使える。Go ではこの「パターンを保持したまま返す」操作に追加の変数宣言が必要になる。

#### 4. Go との定性的比較

| 観点 | MoonBit（本プロジェクト） | Go（同等のコンパイラ実装の場合） |
|------|--------------------------|-------------------------------|
| AST 型定義 | `enum` で 1 ファイルに集約 | インターフェース + 構造体の組み合わせ |
| バリアント追加の安全性 | **コンパイル時に全箇所を検査** | `default:` で見逃しの可能性あり |
| ネスト分解 | `Ptr(Struct(name))` を 1 行で | 2〜3 段の型キャスト必要 |
| タプルパターン | `(target, source)` で同時分解 | 外側 switch + 内側 switch のネスト |
| 束縛 | `as p` でパターン全体を束縛 | 明示的な変数宣言が必要 |
| 可読性 | パターンが宣言的 | 手続き的（if-else チェイン） |
| 行数の目安 | match 1 式で 5〜15 行 | 同等処理に 15〜40 行 |

#### 5. AI エージェントにとっての恩恵

パターンマッチングの網羅性チェックは、AI エージェントにとって特に有益であった。理由は以下の通り:

- **段階的な AST 拡張が安全**: Phase ごとに enum のバリアントを追加する開発スタイルにおいて、コンパイラが「このバリアントが未処理」と教えてくれるため、AI エージェントは漏れなく対応箇所を修正できた
- **コード生成の構造化**: 「各 AST ノードに対して 1 つの match 腕を書く」という明確な構造により、AI エージェントのコード生成が体系的になった
- **リファクタリングの安全性**: Phase 4 で型なし AST → 型付き AST の変換を導入した際、enum の型が変わったことで影響範囲が全て `moon check` で特定でき、大規模なリファクタリングを自律的に完遂できた

### 結論

MoonBit のパターンマッチング × 代数的データ型の組み合わせは、コンパイラの AST 操作において Go のインターフェース + 型スイッチと比較して**安全性・簡潔さ・可読性の全ての面で優位**であった。

特に、73 箇所の match 式と 62 バリアントの enum という規模において、網羅性チェックの価値は単なる便利機能を超え、**段階的に AST を拡張していく開発手法そのものを可能にする基盤**として機能した。Go では `default: panic("unreachable")` で逃げがちな箇所が、MoonBit ではコンパイル時に全て捕捉される。この違いは、6 フェーズにわたる継続的な AST 拡張においてバグの発生を構造的に防いだ。

AI エージェントの自律開発という文脈では、「コンパイラに教えてもらえる」ことの価値はさらに大きい。人間開発者なら記憶や経験で対処できる拡張漏れを、AI エージェントはコンパイラの網羅性チェックに完全に委ねることができた。これは MoonBit の型システムと AI エージェントの自律性が相乗効果を生んだ好例と言える。
